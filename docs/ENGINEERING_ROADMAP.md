# 中短期工程化路线

本文用于把工程化工作拆成可执行批次。目标不是追求技术栈复杂度，而是让后续功能开发、移交开发和长期维护都能在稳定边界内推进。

## 1. 当前基线

| 项 | 当前状态 |
| --- | --- |
| 发布形态 | GitHub Pages 静态部署，无构建工具 |
| 主控文件 | `app.js` 低于 6200 行门槛，主控继续收敛为启动与调度层 |
| 样式文件 | `styles.css` 低于 5500 行门槛，后续新增样式需优先复用或分区 |
| 自动检查 | `npm run check` 覆盖语法、测试、架构、静态页面、smoke 契约和发布卫生 |
| 业务测试 | 核心计算、导入导出、图表 option、页面 view model 已有 Node 测试 |
| 主要残余风险 | `app.js` 仍偏大、真实浏览器自动化不足、CSS 单文件继续增长 |

## 2. 短期计划

建议按 3-5 个稳定提交推进，每个提交都必须保持 `npm run check` 全绿。

| 批次 | 目标 | 交付物 | 验收 |
| --- | --- | --- | --- |
| S1 | 给 `app.js` 留出维护缓冲 | 已迁出项目 seed builder、创建项目表单同步、创建项目表单校验、项目记录工厂、场景管理、对比页、预测版本列表、测算前置校验、结算电量展示、电量派生状态和多组事件绑定 | `app.js` 已降到 6200 行以内，单函数低于 120 行 |
| S2 | 提高页面回归可靠性 | 把静态 smoke 契约扩展为关键流程浏览器 smoke 方案 | 登录、项目、导入、预测、测算主链路可重复验证 |
| S3 | 固化业务变更入口 | 为全口径收入、结果总览、多方案对比补充更多黄金样例 | 业务口径变化时测试先失败，避免只靠人工看页面 |
| S4 | 控制样式增长 | 已给 `styles.css` 增加 5500 行架构门禁，后续补分区目录或拆分计划 | 新页面样式有明确归属，减少选择器互相污染 |
| S5 | 发布与安全收口 | 强化发布前检查、README 使用说明和 Pages 可用性检查 | 外部用户可直接打开工具，仓库不带本地运行痕迹 |

## 3. 中期计划

中期目标是在不打断现有静态部署的前提下，让项目具备团队维护能力。

| 方向 | 判断条件 | 推荐动作 |
| --- | --- | --- |
| 控制器拆分 | `app.js` 仍超过 6000 行，或新增页面要频繁改主控 | 按页面建立控制器模块，`app.js` 退化为启动与调度层 |
| CSS 拆分 | `styles.css` 超过 5500 行，或同类组件出现重复样式 | 先按注释分区，再评估拆为多 CSS 文件 |
| 构建工具 | 模块数量继续增长，脚本顺序成为主要风险 | 评估 Vite + ESM，先做迁移 PoC，不直接大改 |
| 类型约束 | 字段口径频繁变动，测试仍难覆盖结构错误 | 评估 TypeScript 或 JSDoc schema，先从 `src/domain/` 开始 |
| 数据持久化 | 需要多人协作、账号隔离或云端共享 | 设计后端数据契约，再接入真实鉴权与存储 |

## 4. 执行纪律

1. 每个批次先改低耦合模块，再动页面主控。
2. 每次稳定提交前必须运行 `npm run check`。
3. 功能口径变化必须同步业务文档、数据字典或验收用例。
4. 不为了工程化提前引入构建复杂度；只有当前门禁无法继续兜住风险时才升级技术栈。
5. 本地提交和远端 push 分开处理，push 前再确认仓库状态和 GitHub Actions。
